# HtmlTemplate.getCode()/getCodeWithComments() left unimplemented

`HtmlTemplate` has three introspection methods in the real GAS surface: `getRawContent()`, `getCode()`, and `getCodeWithComments()`. Only `getRawContent()` is implemented; the other two remain `GasPNotImplementedError` stubs, deliberately.

`getRawContent()` is unambiguous — Google's docs say it "returns the unprocessed content of this template," i.e. the original template string, scriptlets and all. `getCode()`/`getCodeWithComments()` are different: the docs only say they generate "a string of JavaScript code, based on the template file, that can be evaluated," with no specification of the actual output format. That format is GAS's internal compiled-template representation, not a documented contract.

We verified this empirically rather than guessing: a script was run against a real Apps Script execution (`HtmlService.createTemplate()` with escaped/unescaped/control-flow/comment scriptlets, then logging `getCode()`/`getCodeWithComments()`). The real output showed:

- **A different init call per method.** `getCode()`'s output calls `HtmlService.initTemplateExp()`; `getCodeWithComments()`'s output (wrapped in an `eval()` string) calls `HtmlService.initTemplate()` — no `Exp` suffix. Whether this is a deliberate distinction or incidental wasn't possible to confirm from one sample.
- **An apparent real bug.** A `<?# comment ?>` scriptlet compiles to the bare literal text `# this is a comment, should vanish ;` in the output — not valid JavaScript at that position (a bare `#` is a syntax error in that context). This directly contradicts the docs' own claim that the generated code "can be evaluated." Replicating this would mean deliberately reproducing a bug, not documented behavior.
- **Precise, fragile formatting** with no specified grammar: one output line per *source* line (not per statement), `output._$ = ` for escaped scriptlets vs. `output._ = ` for both literal text and unescaped scriptlets, inconsistent-looking whitespace around semicolons, and real newlines re-encoded as literal `\n` inside string chunks. Reproducing this from a single sample risks getting the general shape right while being subtly wrong on cases not covered by that sample (multi-line scriptlets, empty lines, quotes/backslashes in literal text, CRLF) — with no way to verify correctness short of re-running against real Apps Script for every edge case.

Given the target is an undocumented internal representation (not a stable API), includes at least one apparent bug we'd have to replicate rather than fix, and is debug/introspection tooling that real scripts don't functionally depend on (nothing downstream `eval`s this output in a normal GAS web app), we chose not to guess at it. A wrong-but-plausible-looking implementation is worse than a clear `GasPNotImplementedError` — it would silently pass casual inspection while being unverifiable against the real system it's supposed to mirror.

See [CONTRIBUTING.md's "Filling in a stub" step 2](../../CONTRIBUTING.md#filling-in-a-stub) for the general pattern this follows: when Google's docs don't specify behavior, verify against a real Apps Script execution before implementing; if even that can't establish a stable, verifiable contract, document why and leave it as a stub rather than ship a guess.
