# No automatic browser reload on .gs/.js/.ts source change

We implemented and then rolled back a hot-reload feature (#23): extending Vite's own `server.watcher` to watch `srcDir` and sending `server.hot.send({ type: 'full-reload' })` on any change, so an already-open browser tab would reload automatically after a source edit.

True HMR isn't possible here — `.gs`/`.js` source runs server-side in a Node `vm` sandbox, never in the browser, so there's no client-side module to hot-swap. The only two options are "reload the whole page" or "don't." Full-reload on every change is also coarser than it needs to be: the harness already rebuilds its `vm` context fresh on every request, so an edit to a function only called via `google.script.run` needs no reload at all — the next `google.script.run` call already reflects the change. The reload only pulls real weight when `doGet()`'s returned HTML itself changes, since that HTML is baked into the already-loaded page.

Narrowing the reload to just "did this edit affect `doGet()`'s output" is unsafe without full call-graph analysis: `doGet()` typically calls helper functions defined elsewhere in `srcDir`, and Apps Script permits dynamic dispatch (e.g. `this[fnName]()`), so textually diffing `doGet()` alone risks silently serving stale HTML when a dependency changes but `doGet()`'s own source doesn't. Given that risk, and that reloading on every change unnecessarily discards client-side state for the common RPC-only-edit case, we rolled the feature back rather than ship a heuristic that can be silently wrong. Developers refresh manually; RPC-only edits already need no manual action beyond re-triggering the call.
